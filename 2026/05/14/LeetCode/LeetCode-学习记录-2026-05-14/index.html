💡 对比收获

  • 暴力DFS（起点枚举） vs 前缀和+哈希表：暴力法是正向思维（每个节点为起点向下搜），前缀和是逆向思维（查表找之前的互补和）
  • int 溢出陷阱：节点值范围 [-10⁸, 10⁸]，累加可能超 INT_MAX，要用 long long
  • map 的 key 类型也必须用 long long：否则 curr_sum 从一个 long long 存入 unordered_map<int,int> 时会截断，可能误撞到 map[0]
  • 计数 vs 列路径：前缀和查表只能计数，不能列路径（哈希表只存了次数，丢了位置信息）

[Medium] Path Sum III (437)

🎯 核心思路

前缀和 + 哈希表 + 回溯。一次 DFS 遍历树，用 unordered_map<long long, int> 记录从根到当前节点路径上各前缀和出现的次数。对于当前节点，查表找 curr_sum - targetSum 出现的次数，累加到结果中。递归返回前回溯，移除当前前缀和，确保只统计当前路径上的前缀和。

🔑 关键代码片段

📚 学到的知识点

  • 树上前缀和：从根到当前节点路径上累加，与数组前缀和本质相同，但需要回溯
  • 逆向思维：不是”从当前节点往下找=targetSum”，而是”之前有没有前缀和 = curr_sum - targetSum”
  • **map[0] = 1**：处理根到当前节点整条路径和正好等于 targetSum 的情况
  • 回溯的必要性：不同路径分支互不影响，必须用 map[curr_sum]-- 撤销当前分支的贡献
  • long long 防溢出：节点值范围大，int 可能溢出；且 map key 类型也要用 long long，否则截断可能导致误命中 map[0]

⚠️ 易错点 & 反思

  • ❌ curr_sum 和 need 忘记用 long long → 203 测试用例 int 溢出
  • ❌ map key 类型仍为 int → curr_sum 截断后撞到 map[0]，误加计数
  • ❌ 忘记 map[0] = 1 → 根到当前节点整条路径和为 targetSum 的情况漏统计
  • ❌ 忘记回溯 map[curr_sum]-- → 不同路径分支间的前缀和互相干扰
  • 💡 路径三部曲：112（有没有）→ 113（列路径/根到叶子）→ 437（计数/任意路径）。同个问题逐步升级

🔗 关联题目

  • [113] Path Sum II — 根到叶子，列出所有路径
  • [560] Subarray Sum Equals K — 数组版前缀和+哈希表，思路完全一致

💬 补充讨论：如果要列路径怎么做？

437 要求计数，前缀和查表刚好满足（O(n)）。但如果要列出所有路径，有两种做法：

方向一：双重 DFS（暴力法，O(n²)）

改造 [113] 的 “根→叶子” 为 “任意起点→任意终点”：

做法 时间复杂度 空间复杂度 能列路径？
双重 DFS O(n²) O(n) ✅
前缀和+哈希 O(n) O(n) ❌仅计数

方向二：路径向量 + 前缀和（进阶，O(n)）

  • unordered_map<long long, vector<int>> 存 前缀和 → 出现时的路径长度列表
  • 同时维护 vector<int> path 记录当前遍历路径
  • 找到 curr_sum - targetSum 时，用路径长度索引在 path 中切片，得到具体路径
  • 回溯时同时 pop path 和移除 map 中的长度记录

理论上 O(n) 可列路径，但实现较复杂，通常面试用方向一就够了。
